# PKCE Recap

## Server-Side Configuration (Spring)

**Step 1: Dependency in pom**
Add `spring-boot-starter-oauth2-client` to the pom. This imports the classes needed for OAuth2.

**Step 2: Security filter**
At startup, a security filter is added to the chain. It is configured for Spring boot and will call `oauth2Login()`.

**Step 3: Server-side configuration**
Configure on the server, in Spring's `application.yml`, which authentication server to use and the `client_id` and `client_secret` values.

## The OAuth2 Flow with PKCE

### 1. Initial Request (unauthenticated)

```
GET http://localhost:8080
↓ 302 Redirect
```
Spring detects that the user is not logged in and immediately redirects to:
`http://localhost:8080/oauth2/authorization/github`

### 2. Redirect to GitHub (Authorization Server)

```
http://localhost:8080/oauth2/authorization/github
↓ 302 Redirect to GitHub
```
Spring builds the call with the following elements:

| **AUTHORIZATION REQUEST** |
|---|
| **GET** https://github.com/login/oauth/authorize |
| Headers: (none) |
| **Params:** response_type=code, client_id=Ov23liuXpZ3RQ1whTJSx, scope=read:user, state=q-sqDjq2Nd..., redirect_uri=http://localhost:8080/login/oauth2/code/github, code_challenge=6o0DUie..., code_challenge_method=S256 |
| **Response:** 302 redirect → http://localhost:8080/login/oauth2/code/github?code=46b29...&state=... |

**Parameters:**
- **response_type=code**: I want to receive an authorization code. It's the most secure method.
- **state**: GitHub stores the state and will send it back in the callback (anti-CSRF security) to prevent an attacker from sending a code not initiated by the user.
- **code_challenge**: A code generated by Spring with an internal verifier. In the next step, the verifier is sent in plain text. When Spring requests the token, GitHub first verifies that the verifier is correct before releasing it. Without the verifier, the token cannot be exchanged — this adds security in case an attacker tries to perform a man-in-the-middle attack. The attacker may capture the code, but when sending it back to GitHub without the verifier, GitHub will refuse to issue the token.
- **code_challenge_method=S256**: Algorithm used.

### 3. Login on GitHub

GitHub sees that the user is not logged in and sends them to the login page. Everything happens through successive redirects with status code 302 (HTTP 302, temporary redirect). The user enters username, password and possibly MFA.

### 4. Callback from GitHub to Spring

Once the user logs in, Spring receives:

```
code=6afba5474744a03177f9 &
state=4x0fpyoU6UW-NR3A9qQdLqlOCkChccXqZ0IZcyQPBHU%3D
```

- **code**: The one generated by GitHub.
- **state**: The one previously sent by Spring to GitHub. Extra check (CSRF).

### 5. Server-to-Server (Spring → GitHub)

Then Spring calls GitHub directly server-to-server, without going through the client.
The `code_verifier` and the freshly received `code` are included in the call, making the transaction secure.
GitHub performs the checks: it validates the verifier by recalculating the challenge it had received using `SHA256(code_verifier)`.
The same POST also includes the `client_secret` (in addition to the `client_id`) as a value in the header, which GitHub uses to validate Spring.
GitHub responds with the `access_token`.
Then the server makes another call to GitHub to fetch the user data, passing the access token just received as a Bearer in the header.
GitHub responds with the authenticated user's info.

| **CALL 1** | **CALL 2** |
|---|---|
| **POST** https://github.com/login/oauth/access_token | **GET** https://api.github.com/user |
| Authorization: Basic base64(client_id:client_secret) | Authorization: Bearer gho_XXXX... |
| **Body:** grant_type=authorization_code, code=46b29..., redirect_uri=http://localhost:8080/login/oauth2/code/github, code_verifier=hexYmN4R... | Accept: application/json |
| **Response:** access_token: gho_16C7e42F... | **Response:** login: 2ins, id: 5638192, name: federico frascarelli, email: federico.frascarelli@gmail.com |
